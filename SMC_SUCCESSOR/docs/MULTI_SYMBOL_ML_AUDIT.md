# Multi-Symbol ML Pipeline Audit

## Scope

Audit of `smc_successor/ml/`, `smc_successor/agents/`, `smc_successor/backtest/` and `docs/ML_DATASET_SPEC.md` for multi-symbol readiness, data leakage, determinism, and schema compatibility.

---

## Findings

### 1. Dataset builder supports multi-symbol iteration

**Status**: âś… Implemented

`build_ml_dataset()` in `dataset_builder.py` (line 116) iterates over `config.symbols` and processes each independently. Each symbol produces `data/ml/{symbol}/{schema_version}_{symbol}.parquet`.

**Gap**: No combined multi-symbol output file exists. Training currently requires manual concatenation. A combined `v4_dataset.parquet` merging all symbols is needed.

### 2. No look-ahead bias in label generation

**Status**: âś… Clean

Labels are generated by forward-simulating each trade via `_simulate_trade_with_stats()`. The entry uses the signal bar's close. Future bars are scanned for SL/TP hits. Features are extracted from data up to and including the signal bar only. No future data leaks into features.

### 3. Feature extraction is deterministic

**Status**: âś… Clean

`FeatureEngine.extract_features()` produces identical output for identical input. The dataset builder verifies determinism by re-reading the parquet and checking `df.equals(df2)`.

### 4. Labels generated from correct future context

**Status**: âś… Clean

`_simulate_trade_with_stats()` walks forward from `signal_idx + 1` to `signal_idx + max_hold_bars`. Labels use only data strictly after the signal bar. Exit reason, PnL in R, MFE, and MAE are computed correctly.

### 5. Schema compatibility maintained across symbols

**Status**: âš ď¸Ź Partial

All symbols share the same feature columns because they all go through the same `FeatureEngine`, `build_scalping_context()`, and `AgentOrchestrator.analyze_context()`. However:

- `agent_decision_ml_probability` is 100% NaN (orchestrator receives no ML probability during dataset building). This column is meaningless for training and should be excluded.
- `symbol` column is present and correctly populated per row.

### 6. Missing MT5 data handling

**Status**: âš ď¸Ź Weak

`build_ml_dataset()` has no try/except around `load_frame()` or `build_scalping_context()`. If a symbol's raw data is missing or corrupt, the function raises an unhandled exception. Symbols with empty data produce no output file silently.

### 7. Progress reporting per symbol

**Status**: âš ď¸Ź Minimal

The `progress_cb` parameter is accepted but not called with symbol-level granularity during the signal processing loop. The user sees no indication of which symbol is being processed or how many candles have been consumed.

### 8. Two competing feature lists

**Status**: âš ď¸Ź Divergent

| List | Location | Count | Notes |
|------|----------|-------|-------|
| `DEFAULT_FEATURES_ML` | `ml/train.py:22` | 36 | Still has `risk_multiplier`, `ml_probability`, `ml_threshold`; missing `market_regime`, agent columns |
| `FEATURES_ML_V3` | `ml/trainer.py:24` | 55 | Complete set with all agent+ICT+Wyckoff+structure features; used by trainer and walk-forward |

The `train_from_csv()` function in `train.py` uses the stale `DEFAULT_FEATURES_ML` list. The trainer and walk-forward use the correct `FEATURES_ML_V3` list.

### 9. Per-symbol data quality varies

**Status**: âš ď¸Ź Untracked

No per-symbol quality metrics are computed. Symbol distribution, missing feature ratio, and label balance are not reported. A symbol with 0 samples produces no error, no warning, and no output.
